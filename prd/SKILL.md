# PRD Skill v1

Produces a complete Product Requirements Document that immediately feeds into the module-creation skill for task decomposition. Together they generate all three documents the controller needs to track project progress: PRD (goals), Decomposition (how), Task List (what's next).

## When to use this skill

- User wants to start a new project and needs a plan before building
- User wants to formalize an existing idea before executing
- User wants to update an existing PRD (graceful update path — see below)
- Any time "what are we building" needs a written answer

## Step 1: Intake interview

Ask ALL of these. Can be collected in a single exchange.

```
PRD INTAKE QUESTIONS

1. PROJECT NAME + ONE-LINE DESCRIPTION
   "What are we building? One sentence."

2. PROBLEM STATEMENT
   "What problem does this solve? Who has this problem? How are they solving it today?"

3. SUCCESS CRITERIA (2-4 measurable outcomes)
   "How will we know this project worked? What does done look like?"

4. CONSTRAINTS
   "What are the hard constraints? Time, budget, tech stack, team size, dependencies?"

5. P-ITEMS (high-level goals, 3-8)
   "Break the project into 4-8 major goals. Each one should be independently completable
    and produce something valuable. These become the P-items."

6. WHAT EXISTS NOW
   "What's already built? What can we reuse? What are we starting from scratch?"

7. WHAT'S OUT OF SCOPE
   "What are we explicitly NOT building in this version?"

8. KEY DECISIONS ALREADY MADE
   "Any architectural or product decisions that are locked in?"

9. OPEN QUESTIONS
   "What's still undecided that could affect direction?"
```

Minimum viable intake: questions 1, 3, 5. Fill in reasonable defaults for the rest and flag them as assumptions.

## Step 2: Generate the PRD

Produce a PRD with this exact structure:

```markdown
# [Project Name] — PRD v1.0

**Status:** PLANNING
**Created:** [date]
**Last updated:** [date]

---

## Problem Statement
[1-2 paragraphs: what problem, who has it, how they solve it today]

## Success Criteria
- [ ] [Measurable outcome 1]
- [ ] [Measurable outcome 2]
- [ ] [Measurable outcome 3]

## Constraints
- [Constraint 1]
- [Constraint 2]

## Out of Scope (v1)
- [What we're not building]

## P-Items

### P1 — [Name]
**Goal:** [One sentence — what this accomplishes]
**Status:** TODO
**Depends on:** [other P-items, if any]

### P2 — [Name]
**Goal:** [One sentence]
**Status:** TODO
**Depends on:** none

[continue for all P-items]

## Key Decisions
| Decision | Rationale | Date |
|----------|-----------|------|
| [What was decided] | [Why] | [date] |

## Open Questions
- [ ] [Question 1]
- [ ] [Question 2]

## Build Order
[Dependency-ordered sequence of P-items — which must come first]

---
*PRD v1.0 — generated [date]*
*Decomposition: auto-generated by module-creation skill*
```

## Step 3: Auto-invoke module-creation

**Immediately after generating the PRD**, invoke the module-creation skill in linked mode:

1. Pass the P-items as input to module-creation
2. Module-creation generates:
   - `[project]-decomposition.md` — module breakdown with tasks per module
   - `[project]-tasks.md` — master task list (human-readable)
   - `[project]-tasks.json` — task manifest (machine-readable for controller)
3. All four documents (PRD + 3 decomposition outputs) are saved together

Tell the user: "PRD complete. Running decomposition to generate task list and controller manifest..."

## Step 4: Commit all documents

If the project has a git repo registered:
1. Commit all four documents to `docs/` directory in the project repo
2. Commit message: `docs: initial PRD v1.0 and decomposition [project-name]`
3. If no repo yet, trigger the project registration flow (one question: "Want me to create a GitHub repo for this project?")

## Step 5: Register with controller

Add project entry to the controller's project manifest:
```json
{
  "id": "[project-id]",
  "name": "[Project Name]",
  "prd_path": "docs/[project]-prd.md",
  "decomp_path": "docs/[project]-decomposition.md",
  "tasks_path": "docs/[project]-tasks.json",
  "repo": "https://github.com/rdmilly/[project]",
  "status": "PLANNING",
  "created": "[ISO date]"
}
```

## PRD Update Flow (graceful — works before and during build)

When the user updates a PRD (changes P-items, scope, etc.):

1. **Identify what changed** — diff old vs new P-items
2. **For added P-items** → run module-creation on new P-items only → append to decomposition
3. **For changed P-items** → re-run module-creation on those P-items only → update those sections
4. **For removed P-items** → mark tasks `cancelled` in JSON manifest → preserve audit trail
5. **Done tasks** → never touched regardless of PRD change → progress never erased
6. **PRD version** → bump version number (v1.0 → v1.1 → v2.0 for major changes)
7. **Commit** → commit updated docs with message `docs: PRD v[X.X] update — [what changed]`

PRDs can be updated as many times as needed before or during build. Everything stays in sync.

---

## Session-level vs. PRD-level merging (important)

**Every session end** → session branch pushed, PR marked ready for review, Telegram notification
**PRD completion** (all P-items DONE) → project archived in Helix, historical record created

These are separate. You merge session work frequently (multiple times per project). PRD completion is a milestone marker, not a merge gate.

---

## What the controller does with the output

The controller reads `[project]-tasks.json` to:
- Know which tasks are in flight and which are done
- Auto-update PRD P-item status when all tasks for that P-item are committed
- Surface task status in `mesh_project_status` MCP tool
- Generate PRD status reports
- Know when a project is fully complete

Commit format that the controller parses: `[P2-1.3] Description` where `P2-1` is the P-item and `.3` is the task number within that P-item.

## Quality checklist before delivering

- [ ] Problem statement is specific — not "we need X" but "users can't do Y because Z"
- [ ] Success criteria are measurable — not "works well" but "processes 100 records in <5s"
- [ ] P-items are independently valuable — each one could ship separately if needed
- [ ] Build order respects dependencies — nothing in P3 depends on something in P4
- [ ] Out of scope is explicit — scope creep prevention starts here
- [ ] Module-creation has been invoked and all four documents are ready
- [ ] Documents committed to repo
- [ ] Controller project manifest updated

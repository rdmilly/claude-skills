# Module Creation Skill

**A universal module factory for Claude.** Give it any process — sales pipeline, content workflow, infrastructure management, client delivery, anything — and it produces a complete set of buildable automation modules.

## What This Does

Most automation fails because people automate individual tasks instead of understanding the system those tasks live in. This skill fixes that by working from first principles:

1. **Decompose** any process into its value chain — the sequence of stages that transforms nothing into the desired outcome
2. **Identify friction** at each stage using three targeted questions
3. **Generate modules** — each friction point becomes a concrete, buildable automation module
4. **Map dependencies** — modules that feed each other create compound leverage
5. **Prioritize** — build the high-impact foundation modules first

The result is a complete module architecture: a system of interconnected automation components with clear inputs, outputs, triggers, and build priorities.

## Why This Exists

This skill was created during the design of the MW Development lead pipeline, where we needed to systematically identify every automation opportunity across a sales process. The initial brainstorm produced 20 modules across 8 stages — from lead discovery through pipeline optimization. But the real insight wasn't the 20 modules themselves; it was the *process* that generated them.

The framework is domain-agnostic. The same three friction questions that generated sales modules generate equally valuable modules for content pipelines, infrastructure management, product development, or any other repeatable process. Rather than re-derive the framework each time, we packaged it as a reusable skill.

## The Core Framework

### Value Chain Decomposition

Every process that produces value follows a chain of transformations:

```
[Input State] → Stage 1 → Stage 2 → Stage 3 → ... → [Desired Outcome]
```

Each stage transforms something. Discovery transforms "unknown" into "known." Qualification transforms "known" into "worth pursuing." Each transformation has friction — manual work, missing data, slow decisions. Each friction point is a module.

### The Three Friction Questions

At each stage of the chain, ask:

| Question | Module Type | Example |
|----------|-------------|--------|
| **"What data do I wish I had?"** | Intelligence / Data | Pain Signal Detector, Tech Stack Profiler |
| **"What do I keep doing manually?"** | Automation | Multi-Channel Sequencer, Meeting Booker |
| **"What would change my outcome if I knew it beforehand?"** | Prediction / Insight | Timing Optimizer, Revenue Estimator |

### Compound Chains

Modules don't exist in isolation. Module A's output feeds Module B's input. These dependencies create compound leverage:

- **Fan-out nodes** (one module feeds many) → build first, highest leverage
- **Fan-in nodes** (many modules feed one) → synthesis bottleneck, build well
- **Feedback loops** (output feeds back to earlier stage) → creates learning systems

## Usage

### Installation

Drop the `module-creation/` folder into your Claude skills directory. The skill triggers automatically when users ask about process decomposition, module brainstorming, or automation planning.

### Example Output

A typical decomposition produces:

- **6-10 stages** in the value chain
- **15-25 modules** across all stages
- **3-5 compound chains** (module pipelines within the larger pipeline)
- **A top-5 priority stack** for immediate build
- **3 build phases** (foundation → leverage → optimization)

### Integration with Agent Architecture

Modules generated by this skill are designed to be:

- **Standalone containers/services** with defined inputs and outputs
- **Event-driven** via triggers (scheduled, on-event, on-demand)
- **Observable** — all actions logged for learning loops
- **Composable** — modules connect via event broadcasting (gossip protocol)

This maps directly to the Paving Agent architecture: each module is a unit the agent manages, monitors, and orchestrates.

## Design Philosophy

### Modules, Not Monoliths
Each module does one thing. A module that scores leads doesn't also send emails. This makes modules testable, replaceable, and composable.

### Friction-First, Not Feature-First
The framework doesn't ask "what features would be cool?" It asks "where does the process break down?" This produces modules people actually need.

### Compound Value Over Individual Value
A standalone module provides linear value. Connected modules provide exponential value. The framework explicitly maps these connections.

### Learning by Default
Every module logs its actions through the observer. The epigenetic layer promotes successful patterns to conventions. You teach the system once; it remembers permanently.

## Extending the Skill

To add a new domain template, add it to `references/domain-templates.md` following the existing format:

1. Name the domain
2. List the chain stages in order
3. For each stage, define the transformation
4. List 2-3 common friction points per stage
